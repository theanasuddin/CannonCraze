# Contributing to Cannon Craze

Thanks for taking the time to contribute. This page explains how the project is
put together and what a good contribution looks like.

## Ways to contribute

- **Report a bug**: open an issue with the bug template. Screenshots or a video
  help enormously, as does your OS and how you launched the game.
- **Suggest a feature**: open an issue with the feature template. Explain the
  player problem it solves, not just the mechanism.
- **Send a pull request**: for anything beyond a typo, open an issue first so
  the direction can be agreed before you invest time.

## Project setup

The desktop game is a Processing 4 sketch: install
[Processing 4](https://processing.org/download), open `CannonCraze.pde`, press
Run. There is no build system and there are no dependencies.

The Android app lives in [`android/`](android/) and is a standard Gradle
project; see [`android/README.md`](android/README.md).

## Ground rules for code

- **Everything stays procedural.** The scene, the icons, the sounds: all of it
  is generated by code in this repository. Do not add bitmap or audio assets.
- **The design canvas is 960 x 600.** Gameplay geometry is fixed; the viewport
  extends the world to fill any window or screen. New visuals must respect the
  world-extension helpers (`worldLeft()`, `worldRight()`, `worldTop()`,
  `worldBottom()`) so they look right at every aspect ratio.
- **Keep the two platforms in step.** A gameplay change in the desktop tabs
  needs the same change in `android/app/src/main/java/.../Sketch.java`. The
  sections there mirror the desktop tabs one to one.
- **Match the style you see**: two-space indentation, aligned assignment
  blocks, section banner comments, uppercase tracked UI text, palette colors
  from `Theme.pde` only. No em dashes anywhere, in code or docs.
- **Test before you push.** Run the game, resize the window while playing,
  finish a run, change every setting, and confirm `data/settings.txt` and
  `data/high_score.txt` still round-trip.

## Pull request checklist

- One topic per PR, with a clear description of what changed and why.
- Screenshots or a short clip for anything visual.
- No new warnings in the Processing console.
- Docs updated when behavior changes (README tables, help modal, guides).
